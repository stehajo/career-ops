# QUALITAETSGATE_VORHABENPROJEKT_ROO_V2_1

## Zweck

Dieses Gate entscheidet, ob eine Uebergabe nur interessant klingt oder tatsaechlich anschlussfaehig ist.

## Abgrenzung zu HERMES

Dieses Gate bewertet nur lokale Roo-Uebergaben.
Es ist **keine** HERMES-Checkliste und kein formaler HERMES-Entscheid.

## Musskriterien fuer jede dauerhafte Uebergabe

- Zielrolle klar
- Dokumenttyp klar
- Status klar
- empfohlenes Dateiziel klar
- Kernproblem klar
- Zielbild klar
- Scope klar
- Nicht-Ziele klar
- Risiken oder Blocker benannt
- offene Fragen benannt
- naechster Roo-Schritt genannt

## Zusatzkriterien fuer ARCHITECT_READY

- relevante Randbedingungen vorhanden
- betroffene Bereiche oder Schnittstellen benannt
- Loesungsoptionen oder Trade-offs benannt

## Zusatzkriterien fuer EXECUTION_READY

- enger Auftrag
- betroffene Komponenten oder Dateien genannt
- Definition of Done vorhanden
- Akzeptanzkriterien testbar
- Sicherheitsrahmen vorhanden
- Rueckgabeerwartung genannt

## Zusatzkriterien fuer QA_READY

- Pruefgegenstand klar
- Pruefbasis benannt
- Evidenzbedarf klar
- Trennung zwischen bestanden, fehlgeschlagen und unklar moeglich

## Zusatzkriterien in HERMES-Vorhaben

Wenn das Handoff in einem HERMES-Vorhaben verwendet wird, muessen zusaetzlich sichtbar sein:

- Phase
- Szenario
- Vorgehensweise klassisch oder agil
- betroffene HERMES-Ergebnisse oder Checklisten
- betroffene `LOCAL_ROO`-Zusatzartefakte

## Harte Stopps

Eine Uebergabe ist nicht ready, wenn:
- sie Scope und Roadmap vermischt
- sie Gegenwart und spaeter vermischt
- sie Wunschliste statt Entscheidungsvorlage ist
- sie operative Arbeit freigibt, obwohl Produkt- oder Strukturfragen offen sind
- sie kein Sicherheitsrahmen fuer CODE oder DEBUG enthaelt
- sie mehrdeutig ist, wer eigentlich Empfaenger sein soll
- sie eine HERMES-Freigabe behauptet, ohne dass das zugehoerige HERMES-Artefakt vorliegt

## Kurzregel

Sauber verdichtet ist besser als breit und halbgar.
